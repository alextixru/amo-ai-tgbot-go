# Shadow Tools — Lazy Schema Loading

## Проблема

13 tools с полными JSON Schema — это тысячи токенов контекста при каждом запросе LLM.
Большинство вызовов используют 1-3 tool, остальные 10+ — мёртвый груз.

## Идея

LLM видит 13 tool с **минимальной схемой** (name + description + action).
При первом вызове tool возвращает **полную схему** вместо результата.
При повторном вызове с валидными параметрами — выполняет действие.

---

## Поток вызовов

```
1. LLM получает 13 tools с минимальным Input:
   { action: string }  — для каждого tool

2. Пользователь: "Создай сделку в воронке Продажи"

3. LLM вызывает: entities({ action: "create" })

4. Tool видит: action есть, но остальных обязательных полей нет
   → Возвращает полную схему:
   {
     "schema": true,
     "tool": "entities",
     "action": "create",
     "required_fields": { ... полная схема для create ... },
     "available_values": {
       "pipelines": ["Продажи", "VIP", "Партнёры"],
       "statuses": { "Продажи": ["Новая заявка", "В работе", ...] },
       "users": ["Иван Петров", "Мария Сидорова"]
     }
   }

5. LLM формирует полный запрос и вызывает entities({...}) повторно

6. Tool выполняет действие → возвращает результат
```

---

## Минимальная схема (что видит LLM при старте)

Каждый tool регистрируется с облегчённым Input:

### Для tools без layer (entities, complex_create, files, unsorted)

| Поле   | Описание                              |
|--------|---------------------------------------|
| action | Действие (обязательное)               |

Description содержит краткое перечисление доступных actions.

### Для tools с layer (activities, customers, admin_*)

| Поле   | Описание                              |
|--------|---------------------------------------|
| layer  | Подсистема (обязательное)             |
| action | Действие (обязательное)               |

### Для entities (добавляется entity_type)

| Поле        | Описание                              |
|-------------|---------------------------------------|
| entity_type | leads / contacts / companies          |
| action      | Действие (обязательное)               |

---

## Логика валидации (2 режима работы)

### Режим 1: Schema Request

Условие: пришёл только action (+ layer/entity_type), остальные поля пустые или явно невалидные.

Ответ: JSON с полной схемой для этого action, включая:
- Обязательные и опциональные поля с описаниями
- Доступные значения из справочников (имена воронок, статусов, пользователей и т.д.)
- Примеры значений

### Режим 2: Execute

Условие: пришли обязательные поля для данного action.

Ответ: результат выполнения (как сейчас).

### Граница между режимами

Определяется **по action**: для каждого action известен набор обязательных полей.
Если хотя бы одно обязательное поле отсутствует — режим Schema.

---

## Что входит в Schema Response

### Структура ответа

| Поле              | Описание                                                  |
|-------------------|-----------------------------------------------------------|
| schema            | `true` — маркер что это схема, а не результат             |
| tool              | Имя tool                                                  |
| action            | Запрошенный action                                        |
| description       | Описание что делает этот action                           |
| required_fields   | Объект: имя поля → описание + тип                         |
| optional_fields   | Объект: имя поля → описание + тип                         |
| available_values  | Справочные данные (имена воронок, статусов, пользователей) |
| example           | Пример полного валидного вызова                           |

### available_values — динамические данные

Ключевое преимущество: справочные данные подставляются **в рантайме** из мап CRM-сервисов.

Какие данные для каких tools:

| Tool           | available_values                                        |
|----------------|---------------------------------------------------------|
| entities       | pipelines, statuses, users, loss_reasons, sources, custom_fields |
| activities     | users (для tasks), entity_types (для parent)            |
| complex_create | pipelines, statuses, users                              |
| unsorted       | pipelines, statuses, users                              |
| customers      | users, customer_statuses, segments                      |
| catalogs       | catalog_names                                           |
| products       | catalog_names, product_names                            |
| admin_*        | без dynamic values (работают с именами напрямую)        |
| files          | без dynamic values                                      |

## Description для tools (минимальные, но информативные)

Description — единственное что LLM видит до вызова. Он должен дать достаточно контекста для выбора нужного tool, но не содержать схему полей.

Формат: одна строка — что делает + перечисление actions.

### Примеры

**entities**: "CRUD для сделок, контактов, компаний amoCRM. Actions: search, get, create, update, sync, link, unlink. Вызови с action чтобы получить схему параметров."

**activities**: "Активности привязанные к сущностям. Layers: tasks, notes, calls, events, files, links, tags, subscriptions, talks. Вызови с layer + action чтобы получить схему."

**admin_pipelines**: "Управление воронками и статусами. Actions: search, get, create, update, delete, get_statuses, create_status, update_status, delete_status. Вызови с action чтобы получить схему."

Ключевая фраза в каждом description: **"Вызови с action чтобы получить схему параметров"** — явная инструкция для LLM.

## Реализация в архитектуре

### Что меняется

| Компонент              | Сейчас                        | После                         |
|------------------------|-------------------------------|-------------------------------|
| Input struct (models/) | Полная схема со всеми полями  | Минимальная (action + layer)  |
| Tool registration      | `genkit.DefineTool[FullInput]`| `genkit.DefineTool[any, any]` + `WithInputSchema` |
| Handler                | Валидация → выполнение        | Валидация → схема ИЛИ выполнение |
| Registry               | Без изменений                 | Без изменений                 |
| CRM services           | Без изменений                 | Добавить метод Schema()       |

### Новые компоненты

1. **ShadowInput** — минимальный Input struct для каждого tool (или единый generic)
2. **SchemaResponse** — структура ответа со схемой
3. **SchemaProvider** — интерфейс на CRM-сервисе: возвращает описание полей + available_values для данного action
4. **Shadow middleware / wrapper** — обёртка, которая решает: вернуть схему или вызвать реальный handler

### Где живёт логика

```
app/gkit/tools/
├── shadow.go            — ShadowInput, SchemaResponse, обёртка
├── shadow_schemas/      — схемы полей для каждого tool+action (или генерация из struct)
├── entities.go          — handler остаётся, но вызывается только в Execute режиме
└── ...
```

### Genkit Input: `DefineTool[any, any]` + `WithInputSchema`

Genkit поддерживает `any` как Input type при использовании опции `WithInputSchema`.
Передаём минимальную JSON Schema руками (только action + layer/entity_type где нужно).

На первом вызове LLM шлёт `{action: "create"}` — handler возвращает схему.
На втором вызове LLM шлёт полный плоский JSON — handler парсит в существующий Input struct и выполняет.

**Почему плоский формат:**
- Совместим с текущими Input struct'ами — формат JSON не меняется
- Проще для LLM — все поля на верхнем уровне, без вложенности
- Существующие handler'ы работают без изменений после десериализации

## Экономия контекста (оценка)

| Метрика                     | Сейчас         | Shadow Tools    |
|-----------------------------|----------------|-----------------|
| Токенов на tool definitions | ~500-1500/tool | ~50-100/tool    |
| Итого на 13 tools           | ~8000-15000    | ~700-1300       |
| Schema response (per call)  | 0              | ~300-800 (1 раз)|

Экономия при старте: **~80-90% контекста** на tool definitions.
Цена: +1 round-trip при первом использовании каждого tool в диалоге.

---

## Риски и митигации

| Риск                                     | Митигация                                            |
|------------------------------------------|------------------------------------------------------|
| LLM угадывает формат, не спрашивает схему| Description явно говорит "вызови чтобы получить схему"|
| LLM забывает схему в длинном диалоге     | Повторный невалидный вызов → схема снова             |
| Два round-trip на каждое действие        | Только на первый вызов, далее LLM помнит схему       |
| Genkit не даёт JSON Schema для `any`     | Description компенсирует                              |
| Сложность отладки                        | Логировать режим (schema/execute) в каждом вызове    |

## Поэтапный план внедрения

### Этап 1: Proof of Concept (1 tool)
- Обернуть `entities` в shadow wrapper
- Реализовать SchemaResponse с available_values из entities service
- Протестировать с LLM: получает схему → формирует запрос → выполняет

### Этап 2: Инфраструктура
- SchemaProvider интерфейс на CRM-сервисах
- Генерация SchemaResponse из существующих Input struct (рефлексия или кодогенерация)
- Shadow wrapper как переиспользуемая обёртка

### Этап 3: Миграция всех tools
- Обернуть оставшиеся 12 tools
- Убедиться что description'ы информативны
- Нагрузочное тестирование (latency от +1 round-trip)

### Этап 4: Оптимизации
- Кэширование: если LLM уже получала схему entities.create в этом диалоге, пропускать schema mode
- Частичная валидация: если пришло 80% полей, выполнять а не возвращать схему
- Группировка: schema response может включать схемы нескольких связанных actions
